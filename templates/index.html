  2. Check the capture state.Ready means usable WPA/PMKID material exists. Unusable means there is nothing Hashcat can verify. Needs converter means the converter path must be fixed in Settings.
  3. Open Sources.Add password dictionaries. Add .rule files only for rule-based mutation. Put the most focused sources first with the arrow buttons.
  4. Open Pipeline.Select Ready captures, then run a built-in template or select manual stages and press Start queue.

02

How the pipeline works

A capture contains cryptographic WPA authentication material. A source generates password candidates. A stage defines how those candidates are generated or transformed. A job is one capture combined with one stage. The persistent queue runs jobs in the chosen order.

CAPTUREWhat to verify+SOURCE / MASKWhat to test+STAGEHow to test it→JOBPersistent work unit→RESULTVerified key

There is no instant decryption mode

Every supported method verifies candidates against the capture. Known results may finish instantly because a match already exists in the local potfile. Dictionaries, rules, hybrids and masks differ in how candidates are generated and ordered.

Strategy first vs capture first

OrderSequenceBest when
Strategy firstCreates a separate job for every capture and stage. Stage A runs on each capture before Stage B starts.You want to try the same high-priority method across all remaining networks first.
Capture firstCreates the same separate jobs, but finishes every selected stage on Capture A before Capture B.You want to finish one network before moving on.
Automatic pruningExact duplicate networks are deduplicated. Recovered networks and completed method fingerprints are excluded before enqueue and checked again immediately before GPU launch. Re-uploading the same dictionary under another filename still skips it; changing its content creates a new fingerprint. Captures are never merged into one queue job.

Captures: formats, states and controls

.22000 is Hashcat’s modern WPA format and loads directly. PCAP, PCAPNG and CAP are packet dumps; Handshake Lab converts them locally with hcxpcapngtool. Nothing is uploaded to a cloud service.

StateMeaningNext action

Card controls

  • Click the card to select or deselect it for a manual run.
  • Verify password tests one entered candidate locally with Hashcat. A valid match is saved to Results and redundant queued stages are skipped.
  • Recheck reruns conversion with the configured converter. It does not manufacture missing packets.
  • Diagnostics opens the complete local converter output; the card itself shows only a short explanation.
  • Delete removes the capture, generated hash, converter log and queued history. Recovered results and per-network method memory stay in the audit database.

Incremental capture import

Every Ready import is compared by ESSID and BSSID with networks already stored. A mixed upload keeps only previously unseen networks in its normalized 22000 file. Exact duplicate files and files containing no new networks are rejected; unusable packet dumps remain importable for diagnostics.

Recovered captures and tested-method memory

Capture Database keeps unresolved work at the top and moves cards containing a verified password to the bottom. Green password chips show the matched ESSID and key directly on the card. Expand Tested methods to see every completed Common, Dictionary, Rules, Hybrid or Mask pass, its source, outcome and completion time.

Network-level pruningFully recovered captures are excluded from Pipeline. For unresolved networks, the app removes both recovered hashes and methods already exhausted with the same exact source/configuration. Known results is intentionally never remembered because the potfile can change.

04

Why a PCAP says Unusable

Unusable is a capture-quality result, not a brute-force engine error. Hashcat needs a valid PMKID or sufficient EAPOL message pair. The file in your screenshot reports missing EAPOL M1 and M3 frames plus “no hashes written”, so there is currently nothing to attack.

Common causes

  • Capture began after authentication started or stopped too soon.
  • Only beacons, probes or ordinary data frames were saved.
  • Packet loss removed a required EAPOL frame.
  • A cleanup/filter removed frames required for conversion.
  • The client never performed a fresh authentication while capture was active.

What to do

  1. Keep the original file and open Diagnostics to confirm what is missing.
  2. Capture the authorized network again for longer while a client performs a fresh connection.
  3. Do not filter or clean the file before conversion.
  4. Import the new file and confirm it becomes Ready.
Recheck is not a repair button.It is useful after fixing the converter path. An incomplete packet dump remains incomplete.
05

Candidate sources

Supported dictionaries are .txt, .dic, .dict, .lst and .wordlist. Hashcat mutation rules use .rule.

  • Place router defaults, your own historical patterns and small high-confidence lists before giant generic collections.
  • The shown source order is used by templates that run every dictionary.
  • Remove <8 streams through one dictionary and creates a wpa_... copy without candidates shorter than eight bytes. The original remains untouched, the source switches to the optimized copy, and queued method fingerprints are refreshed.
  • A rule file is not a password list; it mutates words from a selected dictionary.
  • Duplicate files are rejected by SHA-256.
  • Files are copied into this workspace so unplugging the original drive does not break queued work.

Bundled quick sources

The app automatically links Hashcat's small example.dict for engine checks and six bundled rule sets: leetspeak, best66, T0XlC, d3ad0ne, rockyou-30000 and dive. They stay in the local Hashcat installation and appear directly in Pipeline. Start with the small rules; large rules multiply a dictionary into a much larger keyspace.

Cascade deduplication

The Sources action reads connected dictionaries from top to bottom and keeps a global in-memory index. Each candidate is retained only in the first dictionary where it occurs. Processing is streamed line by line into atomic unique_* copies; originals remain on disk, connected entries keep their IDs and order, and .rule files are ignored. The index itself must fit in RAM, so very large combined candidate sets may require substantial memory.

05A

Large wordlists without slow browser uploads

Do not send multi-gigabyte dictionaries through the browser. The browser would need to stream the entire file over local HTTP and may report NetworkError when the request, memory limit or connection is interrupted. Link these files directly from disk instead.

  1. Copy once.Put dictionary files in M:\Handshakes\wordlists. Put Hashcat .rule files in M:\Handshakes\rules.
  2. Open Sources.Press Scan local folder. The server registers each file by its existing local path; it does not upload or duplicate its contents.
  3. Check the list.The filename and size must appear. Use the source arrows to place focused lists before very large generic lists.
  4. Build the queue.Templates using all dictionaries follow this displayed source order.

What happens with very large files

  • A 3.7 GB or 20 GB list can take hours depending on rule expansion, GPU speed and the number of captures.
  • Dictionary + rules can multiply the candidate count many times; test the plain dictionary first.
  • Do not rename, move or delete a linked file while jobs reference it. If you do, jobs become Blocked until the path is restored.
  • Scanning again is safe. Already registered paths are not added twice.
Best orderSmall router defaults and personal patterns first, medium curated sources second, giant generic sources last.
05B

Wordlist Analyzer

Sources → Analyze scans one dictionary; Analyze dictionaries starts every connected dictionary. Rule files are ignored. The scan runs in a background thread, so the web console and GPU queue remain responsive.

MetricDefinitionWhy it matters
Unique WPAFirst occurrence of every candidate containing 8–63 bytes and no NUL byte.Approximate useful keyspace after exact duplicates and length-invalid lines are excluded.
DuplicatesRepeated exact byte sequences anywhere in the same file.Repeated work that cascade deduplication can remove.
Short <8Lines shorter than the WPA minimum.Hashcat mode 22000 rejects them; use Remove <8 to create a filtered copy.
ETAUnique candidates × unresolved networks ÷ the best current/recent measured speed.A planning estimate, not a guarantee; rules and hybrid stages multiply keyspace separately.

Memory and large files

The file is read one line at a time. Exact duplicate fingerprints are stored in a temporary on-disk SQLite table under data\runtime, then deleted after the scan. This avoids loading a 3–10 GB dictionary into RAM. Progress and final metrics are persisted in the main database; changing or filtering the source invalidates the previous analysis.

Byte lengthThe analyzer reports what Hashcat mode 22000 can test locally. A displayed character can occupy multiple UTF-8 bytes, so character count and candidate byte length are not always identical.

What Common passwords means

Common passwords is an attack stage, not a sort mode. It needs no wordlist, rule or other configuration. The app generates a small temporary source for each capture, tests common high-probability candidates first, then local variants based on that capture's SSID and BSSID. A result is displayed only after Hashcat verifies it cryptographically.

Optional by designCommon passwords is disabled in new custom plans unless you explicitly enable it. It is quick, but it is still candidate testing and cannot infer or bypass an unknown key.

Manual queue procedure

  1. Enable only the stages you want.
  2. Choose each dictionary, rule or mask.
  3. Use arrows to set stage order.
  4. Select Ready captures in the launch card.
  5. Select stages, queue order and GPU power.
  6. Press Start queue.

Mask symbols

?d digit, ?l lowercase, ?u uppercase, ?s symbol, ?a common printable characters. Eight digits are ?d?d?d?d?d?d?d?d. Broad masks grow exponentially.

06A

Pattern Builder

Pattern Builder is a generated pipeline stage. It learns reusable structure only from passwords already verified in this local workspace, then ranks exact known keys, case variants, learned stems and numeric/year suffixes, and combinations with the target ESSID. It never sends data online and never claims to derive a key without verification.

  1. Recover or import at least one key.The local Results table is the learning corpus.
  2. Pipeline → Add a stage → Pattern Builder.Drag it after Known results/Common and before large dictionaries.
  3. Enable the stage and queue captures.A compact temporary dictionary is generated separately for each job.
  4. Let method memory prune repeats.The fingerprint includes the Pattern Builder version and the exact local result corpus. The same corpus is skipped; a newly recovered key automatically creates a new method fingerprint.
Candidate capThe generator is capped at 250,000 valid WPA candidates to remain a fast, focused pass. It is a probability aid, not unrestricted brute force.

Build the queue

  • Click a capture chip or stage chip to include or exclude it. All selects everything in that column.
  • The summary is exact: selected captures multiplied by selected stages equals the number of jobs created.
  • Capture first completes all selected stages for one capture before moving to the next capture.
  • Stage first applies the first stage to every selected capture, then advances to the next stage.
  • Balanced is cooler, High is the everyday setting, and Maximum is intended for a monitored overnight run.
  • Start queue is disabled until at least one capture and one stage are selected.

  2. Test cooling.Run a short High job and confirm temperature and fan values are stable.
  3. Queue once.Select a template and captures. Avoid submitting the same plan repeatedly.

Editing and cleaning the queue

  • Six-dot handle: available on waiting jobs only. Drag a job above or below another waiting job; the saved order survives page refresh and restart.
  • Log button: opens the exact Hashcat command, application error and complete process output.
  • Trash button: removes one waiting or finished job together with its job log, telemetry file, output and restore file. Active jobs must be cancelled first.
  • Clear finished: removes Complete, Failed, Blocked and Cancelled history while keeping queued/running work and recovered results.

Closing the browser does not pause the queue. One GPU worker is recommended for predictable maximum throughput.

Pause all, GPU profile and CPU profile

  • Pause all: immediately suspends the active Hashcat process at the operating-system level and persists a global queue hold. GPU utilization may take a few seconds to drain already submitted work, but the current job no longer advances.
  • Resume all: resumes the suspended process, or releases the hold so the next waiting job can start.
  • W1 Low: lowest desktop impact.
  • W2 Gaming: recommended while playing or doing GPU-accelerated desktop work.
  • W3 High: normal fast mode.
  • W4 Maximum: unattended maximum throughput and highest desktop impact.
  • CPU Off: GPU only.
  • CPU Low / Balanced / High: enables a detected OpenCL CPU device and limits the process to approximately 25%, 50%, or all logical CPU cores. The profile updates waiting jobs; an already active job keeps its current backend set.
CPU backend requiredHashcat can use a CPU only when hashcat -I reports an OpenCL CPU device. The coordinator shows the real detection result. A LAN worker can still use its own CPU backend even when the coordinator has none.
Changing W during a running jobHashcat cannot change --workload-profile interactively. The app requests a checkpoint, updates the single profile value in the restore data, and resumes the same session from that checkpoint. This can take a short time while Hashcat reaches a safe restore point.

Network priority and live Queue sort

  • Likely fastest first: combines factory-looking SSIDs, simple readable names and capture size. It estimates which captures are worth trying first; it does not predict the actual password.
  • Factory SSID first: prioritizes recognizable router/vendor names such as TP-Link, ZTE, Keenetic, HUAWEI, Tenda and similar prefixes.
  • Simple SSID first: prioritizes short readable network names over random-looking identifiers.
  • Fewest networks first: runs smaller capture groups first.
  • SSID A–Z, Newest, Oldest: deterministic organizational sorts.
  • Current selection order: preserves the capture order shown when a new queue is built.

10

GPU telemetry explained

  • Speed: verifications per second (H/s, kH/s, MH/s or GH/s).
  • Temperature: current GPU core temperature; Hashcat receives the configured abort limit.
  • Load: compute utilization. Near 100% is normal under load.
  • VRAM: used and total dedicated memory.
  • Power: current board draw and power limit.
  • Clock / fan: graphics clock and fan percentage when exposed by the driver.
  • Progress: completed candidates divided by total reported keyspace.
  • Elapsed / remaining: elapsed wall time and an approximate remaining duration calculated from Hashcat ETA, current keyspace speed or historical runs of the same stage.

Samples are persisted every five seconds. A blank field often means no active workload or unsupported driver telemetry, not an automatic failure. Queue-wide runtime is an estimate: recovered networks, thermal throttling and different captures can shorten or lengthen it.

Reading the chart

The cyan line uses the left GPU LOAD % axis. The red line uses the right TEMPERATURE °C axis. The dashed red line is the configured thermal abort limit. Hover anywhere over the graph to see the exact timestamp, temperature, load and cracking speed for the nearest persisted sample.

10A

Two computers on one LAN

The main PC remains the coordinator: it owns SQLite, captures, queue order, Results and method memory. The second PC runs start-worker.bat, claims one waiting job at a time and sends only progress and verified output back. Both GPUs may work simultaneously on different jobs.

1. Prepare the main coordinator PC

  1. Find its private IPv4 address.Open PowerShell and run ipconfig. Use the active Ethernet/Wi-Fi address, for example 192.168.1.10.
  2. Open Settings.Set Listen address to 0.0.0.0, enable Accept authenticated workers, keep port 8787, then Save. A long random token is generated if the field was empty.
  3. Restart the coordinator once.Run stop.bat, then start.bat, because the listen address is applied when the web server starts.
  4. Allow only the private LAN.In an Administrator PowerShell run New-NetFirewallRule -DisplayName "NewFPV Handshake Lab" -Direction Inbound -Action Allow -Protocol TCP -LocalPort 8787 -Profile Private. Do not enable it on Public profiles.
  5. Test from PC 2.Open http://MAIN_IP:8787/health. It must show {"ok":true}.

2. Prepare candidate sources

Large dictionaries are intentionally not copied through the coordinator API. Use either option:

  • Local mirror: copy the same dictionaries/rules to PC 2. Paths may differ; content fingerprint and size must match.
  • Read-only SMB share: share a folder on the main PC, grant PC 2 read access, then use a UNC root such as \\MAIN-PC\HandshakeWordlists. Wired Ethernet is preferable for very large sources.

Do not edit a dictionary while either PC is running it. A changed file intentionally fails fingerprint matching instead of silently testing the wrong source.

3. Configure PC 2

  1. Extract the portable archive.Use a short local path, for example C:\NewFPV-Handshake-Lab.
  2. Run start-worker.bat.On the first run it bootstraps Python/Hashcat if needed, creates lan-worker.json from the example and opens it in Notepad.
  3. Edit the JSON.Set coordinator_url to http://MAIN_IP:8787, paste the token from main-PC Settings, choose a unique worker_name, and list local or UNC folders in wordlist_roots.
  4. Run start-worker.bat again.The console must say Connected. Settings on the main PC will show the worker as Idle or Running.

4. Queue and operate

  • Build a normal queue. The local Runner and each LAN worker atomically claim different waiting jobs.
  • Overview shows how many workers are online. Queue shows separate telemetry, CPU profile, W1–W4 profile and Pause controls for every GPU lane.
  • Pattern/Common candidate files and normalized 22000 input are sent automatically; dictionary/rule stages resolve an exact content fingerprint on PC 2.
  • Pause all controls every lane. A lane Pause affects only that computer; changing a remote CPU/GPU profile while it is busy applies from its next job.
  • If a worker disappears longer than the configured heartbeat timeout, its job is safely returned to Queued. Duplicate start-worker.bat launches are rejected locally and the coordinator also enforces one active job per worker identity.
  • Verified keys are imported only by the coordinator and immediately enter Results/potfile/method memory.

11

Recovered results

Only cryptographically verified matches are shown. Each row includes ESSID, BSSID, password, recovery method and time. Use Copy for one password or Export recovered.csv for all results.

  • Results are committed to SQLite.
  • Each match is immediately flushed to data\recovered.csv.
  • The downloadable recovered.csv keeps PWMenu-compatible task columns and embeds the colon-formatted BSSID in task.
  • Hashcat stores verified pairs in data\hashcat.potfile.
  • Deterministic fingerprints prevent duplicate result rows.

13

Getting maximum useful speed

  • Use .22000 input where possible to skip conversion.
  • Keep GPU workers at 1; concurrent Hashcat processes compete for the same GPU.
  • Run known results, focused dictionaries, broad lists, rules, then masks.
  • Use Maximum workload only unattended; High keeps the desktop more responsive.
  • Hashcat 7.1.2 uses its pure kernel for WPA mode 22000 even when -O is requested. The process log explicitly reports the fallback, so older optimized benchmark figures are not a direct target for a real 7.1.2 job.
  • Compare performance only after the queue is empty, with the same Hashcat version, driver, workload profile and GPU. Close games, video editors and other GPU-heavy applications before benchmarking.
  • Candidate quality and order matter more than an impressive raw H/s number.
  • Avoid universal ?a masks unless length and structure are tightly constrained.
  • Keep cooling clear and set a conservative abort temperature.
